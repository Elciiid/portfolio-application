Project Overview

The MP3 Streamer is a centralized audio broadcasting system designed for local network deployment. It allows a "Host" to control playback across multiple "Client" devices (speakers) in real-time, supporting scheduled chimes, playlists, and user management.

Core Technologies

  • Backend: PHP 8.x
  • Frontend: HTML5, Tailwind CSS, Vanilla JS
  • Database: SQL Server (PDO), JSON Files
  • Audio: HTML5 Audio API (HTTP 206 Support)

Key Features

  • Sub-200ms audio synchronization
  • Scheduled chimes (Alarms)
  • Biometric-based user authorization
  • Remote restart & device monitoring

System Architecture

The system follows a Server-Authoritative Synchronization model. The Host updates a central state, and Clients poll for that state, adjusting their local playback to match the server's master clock.

API Reference

api_state.php

Core synchronization endpoint.

GET ?client_id=ID&name=NAME

Clients poll this to get master playback state and chime triggers.

POST action=play|pause|seek|volume

Host sends commands to update the global state.

stream.php

Streams MP3 files supporting HTTP 206 Partial Content (Range Requests).

Critical for seeking within large tracks and reducing buffering latency.

Synchronization Workflow

How the distributed broadcast stays in lock-step — from host command to speaker output.

1

Host Playback Control

HOST → SERVER

The master device issues a POST to api_state.php with an action (play, pause, seek, volume). This is the single authority for all playback transitions — no client can alter global state unilaterally.

2

Server Timestamp Authority

SERVER WRITE

On every play/seek action, the server records start_ts — a Unix float timestamp of when playback began at position 0. This value is written atomically to state.json and becomes the master clock reference for all clients.

state.global.start_ts = microtime(true) - seek_offset
3

Client Sync Calculation

CLIENT POLL

Speaker clients GET poll api_state.php every second. The response includes server_time and start_ts. The client resolves the expected audio position, accounting for measured round-trip latency:

expectedTime = (server_time + latency) − start_ts
4

Drift Correction

CLIENT ADJUST

The client compares expectedTime against its local audio.currentTime. Correction is tiered to avoid jarring skips:

  • < 150ms — no action, within tolerance
  • 150ms – 1s — nudge playbackRate (0.95× – 1.05×)
  • > 1s — hard seek via currentTime

Target: sub-200ms synchronization across all speaker zones.

5

Remote Monitoring

OPERATOR VIEW

Each client heartbeat updates state.clients[device_id].last_seen on every poll. The host dashboard reads this to display live device status — online/offline, zone name, and current playback state — across all factory speaker zones without requiring a direct connection to the client device.
