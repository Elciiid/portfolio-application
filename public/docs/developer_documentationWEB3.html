AI Web3 Automation Assistant

AI Web3 Automation Assistant is a premium blockchain operations workspace. It lets users monitor wallets, create automation rules with natural language, ingest real blockchain activity, generate AI transaction intelligence, and receive in-app or Telegram automation alerts.

Product Purpose

A focused AI infrastructure workspace for on-chain monitoring and automation. It is not a swap app, trading bot, NFT product, or generic dashboard.

Operational Flow

Wallet refresh leads to transaction ingestion, AI summaries, rule evaluation, notification creation, and optional Telegram digest delivery.

System Architecture

The app uses a layered Next.js architecture. React screens call lightweight hooks and API helpers. API routes validate input and authenticated user context. Service modules own business logic and call Supabase, Alchemy, Gemini, and Telegram from the server side.

Database and Storage

Supabase Postgres stores user-owned app data. Row Level Security is enabled on the new app tables. Users can only read and mutate their own profile, wallets, automations, transactions for their wallets, and notifications.

Table Description Important Fields
profiles One profile row per Supabase Auth user. id, email, full_name, avatar_url, created_at
wallets Monitored wallet records owned by a user. address, label, chain, balance_usd, enrichment_status, token_summary_json
automation_rules Automation rules parsed from natural language and saved by the user. raw_prompt, condition_type, condition_value, token, action_type, status
transactions Provider-ingested wallet transaction records. hash, type, token, amount, from_address, to_address, ai_summary
notifications In-app automation alert records linked to wallet, rule, and transaction context. title, message, type, read, automation_rule_id, transaction_id

Key Deduplication Rules

transactions: unique per wallet_id + hash
notifications: unique per automation_rule_id + wallet_id + transaction_id
automation execution: evaluates only newly inserted transactions during enrichment

API Reference

Wallets

GET /api/wallets - list signed-in user's wallets.

POST /api/wallets - create a wallet and attempt enrichment if Alchemy is configured.

DELETE /api/wallets/[id] - delete a wallet owned by the user.

POST /api/wallets/[id]/refresh - refresh a wallet, ingest transactions, summarize, and evaluate automations.

Automations

GET /api/automations - list saved rules.

POST /api/automations - create a rule from structured parsed fields.

PATCH /api/automations/[id] - edit rule fields or status.

DELETE /api/automations/[id] - delete a rule.

Transactions and Notifications

GET /api/transactions - list transaction rows, optionally filtered by walletId.

GET /api/transactions/[id] - fetch one transaction detail.

GET /api/notifications - list notifications.

PATCH /api/notifications/[id] - update notification read state.

AI, Demo, and Scheduled Monitoring

POST /api/ai/parse-rule - parse natural-language automation prompts through Gemini with deterministic fallback.

POST /api/demo/monitoring-run - protected Settings button route that runs one monitoring cycle for the signed-in user's wallets.

GET /api/cron/monitor-wallets - scheduled monitoring route protected by CRON_SECRET.

Service Layer

Service Responsibility
walletService.ts Wallet list, create, fetch, and delete data access.
walletEnrichmentService.ts Owns wallet refresh orchestration: Alchemy snapshot, transaction insert, Gemini summaries, and automation evaluation.
automationService.ts CRUD operations for saved automation rules.
automationExecutionService.ts Evaluates active rules against newly ingested transactions and creates notifications.
aiRuleParserService.ts Converts natural language prompts into validated automation rule objects.
aiTransactionSummaryService.ts Generates concise transaction intelligence for new transactions.
scheduledMonitoringService.ts Loads monitored wallets and runs enrichment safely across a scheduled or manual demo cycle.
telegramNotificationService.ts Sends compact Telegram channel digests without blocking in-app notification creation.

Core Workflows

The app's most important behavior is backend-assisted monitoring: refresh wallets, ingest transactions, generate AI context, evaluate automations, and notify operators.

Wallet Monitoring Cycle

sequenceDiagram participant U as User or Cron participant API as Next.js API Route participant S as scheduledMonitoringService participant W as walletEnrichmentService participant A as Alchemy participant G as Gemini participant DB as Supabase participant T as Telegram U->>API: Trigger monitoring run API->>S: runScheduledMonitoring() S->>W: enrichWallet() W->>A: Fetch balances and transfers W->>DB: Store wallet snapshot and new transactions W->>G: Summarize new transactions W->>DB: Save ai_summary W->>DB: Load active automation rules W->>DB: Create matching notifications W->>T: Optional digest delivery API-->>U: Run summary
1

Natural-Language Rule Parsing

AI PARSER

The user enters a prompt such as Notify me when wallet sends over 1000 USDT. /api/ai/parse-rule attempts Gemini structured parsing, validates the schema, and falls back to deterministic parsing if the model fails.

2

Rule Persistence

SUPABASE WRITE

The structured rule is saved to automation_rules. Supported MVP rule types are transfer_amount, receive_amount, and token_movement.

3

Blockchain Enrichment

ALCHEMY

Wallet creation, wallet refresh, demo monitoring, and scheduled monitoring can call Alchemy to fetch balances, token metadata, and recent transfers. Transfers are normalized before insertion into the app database.

4

Automation Execution

RULE ENGINE

Newly inserted transactions are compared against active automation rules. Matches create deduplicated in-app notification rows linked to the rule, wallet, and transaction.

5

Notification Delivery

IN-APP + TELEGRAM

Notifications always persist in Supabase first. Telegram digest delivery is optional and failure-safe: Telegram errors are logged but do not break wallet refresh or in-app alerts.

Quota Protection

Keep scheduled monitoring disabled unless testing. The protected Settings demo run can trigger one controlled cycle for the signed-in user.

Telegram Behavior

Background Telegram delivery respects ENABLE_TELEGRAM_NOTIFICATIONS. The Settings demo route can force Telegram delivery when bot credentials are configured.

Security Notes

Authentication

  • Supabase email/password auth powers sign-in and sign-up.
  • Protected routes redirect logged-out users to /sign-in.
  • API routes use authenticated server context before calling services.

Database Protection

  • RLS is enabled on app tables.
  • User-owned rows are scoped through auth.uid().
  • Service-role access is reserved for server-only scheduled/demo monitoring.

Roadmap and Boundaries

Currently Real

  • Supabase Auth and protected routes.
  • Wallet CRUD and provider enrichment.
  • Blockchain-derived transaction ingestion.
  • Gemini rule parsing and transaction summaries with fallbacks.
  • Synchronous automation execution after ingestion.
  • In-app notifications and Telegram digest delivery.

Still MVP / Future Work

  • No production blockchain indexer or websocket monitoring.
  • No queue or distributed worker infrastructure.
  • No Slack, email, webhook, or push delivery.
  • Wallet risk scoring and 24h change are simplified placeholders.
  • No organization/team/role management yet.
