Developer Case Study / Applied AI Security

Facebook Phishing Detector

A thesis-built anomaly detection system that identifies suspicious phishing links inside Facebook feed posts through a browser extension, hybrid AI scoring, and transparent user controls.

Project Type Browser Extension + AI Backend
Dataset Scale 1.32M URLs
Public Test 397,203 URLs
Model Result F1 0.9308

01 / Context

Problem

Facebook remains a common target for phishing because links can be distributed through posts that look familiar, social, or trustworthy. Traditional blacklist approaches are limited because they depend on known malicious URLs and require constant manual curation. The goal of this project was to detect suspicious links in Facebook feed posts in real time while preserving user control.

The system was designed as an Anomaly Detection System (ADS): instead of only matching known phishing URLs, it models suspicious URL behavior through lexical anomaly scoring and graph-based domain relationship analysis.

02 / System Overview

What The System Does

User-Facing Behavior

  • Monitors Facebook feed posts through a desktop browser extension.
  • Extracts URLs from the Facebook DOM as users browse.
  • Blurs posts that contain suspected phishing links.
  • Provides View Anyway, Report, and Show Link Details controls.
  • Lets users submit feedback for false positives, false negatives, and confirmed detections.

Engineering Behavior

  • Sends extracted URLs to a backend over secure HTTPS REST endpoints.
  • Transforms URLs into lexical feature vectors.
  • Scores lexical anomalies using an Autoencoder trained on benign URLs.
  • Uses precomputed GCN outputs for domain-level structural signals.
  • Stores curated user feedback in Google Firestore for future retraining.

03 / Architecture

Detection Pipeline

1 DOM Scan

Browser extension extracts URLs from Facebook feed posts.

2 REST Request

URLs are sent to the backend through secure HTTPS API calls.

3 Feature Extraction

URL length, domain depth, symbols, shortening, and related lexical features are normalized.

4 Hybrid Scoring

Autoencoder anomaly score and GCN structural score are fused into a final risk signal.

5 UX Response

Suspicious posts are blurred with reporting, override, and link-detail controls.

Runtime Components

  • Browser extension: user-side DOM monitoring, post blurring, overlay controls, reporting UI.
  • Backend API: prediction endpoints, batch prediction, reporting, and review queue handling.
  • Model layer: Autoencoder, GCN artifacts, fusion logic, optional supervised classifier override.
  • Feedback store: Firestore stores curated user labels for future model refinement.
  • Deployment: Dockerized backend hosted on DigitalOcean App Platform with HTTPS.

04 / Model Design

Hybrid Autoencoder + GCN Approach

Autoencoder

The Autoencoder was trained on benign URLs to learn normal lexical patterns. During inference, suspicious links are identified by reconstruction error: URLs that deviate from learned benign structure receive higher anomaly scores.

  • Models normal URL structure rather than memorizing phishing examples.
  • Uses lexical features such as URL length, domain depth, and special-character counts.
  • Features are scaled with RobustScaler to reduce outlier impact.

Graph Convolutional Network

The GCN models structural relationships between domains and recurring infrastructure patterns. This helps surface coordinated phishing behavior that single-link lexical analysis can miss.

  • Uses graph-style domain relationship signals.
  • Precomputes outputs offline for low-latency runtime lookup.
  • Loads artifacts such as gnn_probs.npy and post_node_map.json at backend startup.
The original fusion design combined lexical and structural signals. The thesis describes a configurable fusion layer and reports final tuning where the GCN contributed significantly to the final classification result.

06 / Evaluation

Model And User Testing Results

Evaluation Result Interpretation
Public test F1-Score 0.9308 Strong balance between precision and recall on a large held-out URL set.
Public test Accuracy 0.8753 Generalizable performance across 397,203 public-test URLs.
Reported Precision 99% Most detected phishing links were actually malicious in the reported evaluation.
Reported Recall 87% The system captured most phishing links while preserving conservative user controls.
Reported ROC-AUC 97% Strong separation between phishing and benign URL classes across thresholds.
User test group 30 users + 1 expert Extension usability, warnings, blurring, trust, and perceived safety were evaluated.
The model intentionally favors user safety in some validation contexts. The thesis reports perfect recall on 300 user-reported links but lower precision, meaning the system can blur more aggressively to reduce missed threats. The browser extension offsets this with transparent override and reporting controls.

07 / Extension UX

Safety Without Removing User Control

A security tool can fail if it blocks users without explanation. This extension uses blurring as an intermediate safety layer rather than a hard block. Users can inspect details, report feedback, or continue if they choose.

  • Potential unsafe link detected: suspicious posts are blurred in the feed.
  • View Anyway: users can override the blur while retaining awareness.
  • Show Link Details: detected URLs are shown so users can inspect destinations.
  • Report as phishing: feedback is stored for later review and model improvement.
  • Not malicious: users can flag false positives for curation.

08 / Deployment

Backend Deployment And Runtime Constraints

The backend was containerized with Docker and deployed on DigitalOcean App Platform. DigitalOcean provided a public HTTPS endpoint and SSL certificate management, allowing the browser extension to transmit URL classification requests securely.

A key performance decision was to avoid live graph computation during each prediction request. Instead, GCN outputs were generated offline and loaded as artifacts at backend startup, preserving structural awareness while keeping runtime scoring suitable for a browser-extension workflow.

09 / Limitations

Scope Boundaries

  • The extension focuses on Facebook feed posts in desktop browsers.
  • Facebook Stories were excluded because their DOM behavior is dynamic and harder to intercept reliably.
  • Messenger support was outside the thesis scope and would require stronger privacy controls.
  • The current system analyzes URLs, not full post text, images, or social-engineering language.
  • User feedback is curated for future retraining rather than applied automatically in real time.

10 / Future Improvements

Next Technical Directions

  • Add NLP analysis for surrounding post text to detect urgency, impersonation, and social-engineering cues.
  • Extend detection to Messenger with explicit consent and privacy safeguards.
  • Investigate mobile-compatible deployment through a mobile app or browser-supported extension model.
  • Create a platform-agnostic API for phishing detection outside Facebook.
  • Add richer model monitoring around false positives, false negatives, and threshold drift.
